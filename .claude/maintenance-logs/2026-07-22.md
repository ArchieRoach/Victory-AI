emote-picker toggle, send button) was 36px (`w-9 h-9`) with only a `title` attribute and no

## Task 2: Security Audit

Full re-read of `backend/server.py` (4250 lines) against all prior maintenance logs to avoid
re-flagging fixed issues (CORS allowlist, auth coverage, existing rate limits, AI-token quotas,
no hardcoded secrets, no raw Mongo interpolation, and the already-moderated free-text fields were
all re-confirmed still correct, not re-reported).

**Fixed — Stripe checkout open redirect via unvalidated `origin_url`:**
`/api/payments/checkout`, `/api/tokens/purchase`, `/api/streams/{id}/gift-sub`, and
`/api/ads/checkout` all spliced a client-supplied `origin_url` straight into Stripe's
`success_url`/`cancel_url` with no validation — a user could redirect the post-payment browser
(carrying a real Stripe `session_id`) to an attacker-controlled domain. Added
`_safe_checkout_origin()`, which only allows the value through if it matches the existing CORS
origin allowlist, falling back to the first allowed origin otherwise. Wired into all 4 endpoints.

**Fixed — ElevenLabs `voice_id` path injection:**
`/api/tts/generate` spliced `TTSRequest.voice_id` unsanitized into
`https://api.elevenlabs.io/v1/text-to-speech/{voice_id}` with no format check, letting a
crafted value change which ElevenLabs endpoint the server's own API key hits. Added a
`[A-Za-z0-9]{1,64}` allowlist check, rejecting anything else with a 400.

**Fixed — gym name/description skipped content moderation:**
Every other comparable free-text field (post captions, comments, competition/stream
title/description) is checked via `is_content_flagged`; `POST /api/gyms`'s `name`/`description`
were not, despite being shown publicly via `/api/gyms`, `/api/gyms/leaderboard`, and
`/api/gyms/{gym_id}`. Added the same moderation check (gyms have no update endpoint, so the
create-path fix is complete).

**Fixed — unescaped user input in admin feedback email:**
`POST /api/feedback`'s notification email escaped `type_label` but not `doc['user_name']`,
`doc['user_email']`, `data.page`, or `data.message` before interpolating them into the HTML body
sent to `ADMIN_EMAIL` — the omission looked like an oversight next to the one field that was
escaped. Added `html.escape()` to all four.

`python3 -m py_compile backend/server.py` passes.

No new `console.error`/`console.warn` sites (same 8 as prior days, all previously reviewed as

**Fixed — 3 empty `catch {}` blocks silently swallowing user-triggered action failures:**

  Added a `toast.error`.

  to like-button clicks, no optimistic update) had bare `catch {}` blocks — a failed request left

  both.

**Noted, not fixed (lower severity):** `HomePage.jsx`'s `toggleComments` also has a bare
`catch {}` around the comments fetch, but the panel still opens either way and a failed fetch is
only cosmetically indistinguishable from "no comments yet" — not a stuck or broken state like the
three above, left for a future pass if it needs equal treatment.
